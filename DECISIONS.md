# Decisions log

Running log of implementation choices made outside the spec. One entry per phase
(can have multiple bullets). Format: short rationale, no walls of text.

## P0 — skeleton

- **Build backend**: `hatchling` (modern PEP 517, simple, no extra config).
- **Dependency manager**: `uv` (fast, lockfile-based; project requires it per
  spec §3).
- **Pinned dependency versions**: hard-pinned for reproducibility (`==`); will
  bump deliberately later. All resolved via `uv sync` against PyPI.
- **JWT library**: `pyjwt` (simpler than `python-jose`, no JWE needs in V1).
- **Layout**: single `lcloud` package, sub-packages per domain (`api`, `db`,
  `crypto`, `userbot`, `workers`, `auth`, `utils`). `tests/` mirrors it as
  needed.
- **Alembic `target_metadata`**: left `None` in P0 — wired up in P1 once SQLA
  models exist.
- **No CORS in P0**: backend serves built SPA from same origin in production;
  during dev the Vite proxy will forward `/api/*` to `127.0.0.1:8787`.
- **HTTP host**: bound to `127.0.0.1` only. Public exposure (TLS, reverse
  proxy) is deferred — domain isn't ready yet (see goal.md §2).
- **Python target**: `>=3.11` (spec §3); machine has 3.12 — used as-is.

### Spec amendments captured at end of P0

User added three clarifications before P1 starts (now codified as
"SPEC AMENDMENTS" at the bottom of `goal.md`):

- **Admin = Telegram user_id**, configured via `LC_ADMIN_TG_ID` env var.
  `goal.md` §8 magic-link auth dropped in V1.
- **Telegram login moves into the web UI** (phone → code → optional 2FA),
  replacing the CLI `lcloud-login` flow from §7.1. CLI kept as a fallback.
- **Real credentials provisioned in `.env`** (gitignored): `TG_API_ID`,
  `TG_API_HASH`, `LC_ADMIN_TG_ID`. `.env.example` updated with the new
  field. `lcloud/config.py` gained `lc_admin_tg_id: int` (default 0 =
  "refuse all logins / needs setup").
- **Wrong-account session policy**: archive, not delete. Session files
  belonging to a non-admin Telegram account are renamed to
  `data/session.rejected.<unix_ts>.lcloud` (mode 600) for forensic
  review and never auto-loaded. No automatic cleanup in V1.

## P1 — login + keys + DB

- **DB layer** (`lcloud/db/{base,models}.py`): SQLAlchemy 2.x async with
  `DeclarativeBase` + `Mapped[T]` typed columns. WAL + foreign-keys + 5s
  busy-timeout pragmas wired via `event.listen(..., "connect", ...)` on the
  sync side of `AsyncEngine`. Module-level singleton (`init_engine`,
  `get_sessionmaker`) chosen over DI containers — ergonomics, not religion.
- **Models** match `goal.md` §10 exactly. `mime` defaults to
  `application/octet-stream` (Telegram defaults to that for binary uploads).
  `file_tags` uses `ondelete=CASCADE` so file/tag deletion cleans the
  many-to-many table without app-level bookkeeping.
- **Alembic 0001_initial**: hand-written, not autogenerated, because FTS5
  needs raw SQL anyway (`CREATE VIRTUAL TABLE files_fts USING fts5 ...`)
  plus three sync triggers (insert/update/delete on `files`).
- **Settings → Alembic**: `lcloud/db/bootstrap.py` injects the active
  `Settings` into the alembic config via `cfg.attributes["lc_settings"]`.
  `alembic/env.py` reads it (caller-provided wins over the global
  `get_settings()` cache). This avoids the gotcha where `env.py` would
  always rewrite the URL to the cached production value, even when called
  with a different config (e.g. tests pointing at a tmp DB).
- **Crypto** (`lcloud/crypto/{keys,sign}.py`): PyNaCl Ed25519. Private key
  written via `os.open(O_EXCL) + os.fchmod(0o600)` to eliminate the
  permission-leak window between create-and-chmod. Signature payload is
  the canonical concat from `goal.md` §5 with int64 LE encoding (chosen
  over textual JSON for size + stability).
- **Lifespan order**: `ensure_runtime_dirs` → `ensure_admin_keypair` →
  `init_engine` → `run_migrations` (sync, offloaded via `asyncio.to_thread`)
  → `ensure_admin_owner`. Anything failing here aborts startup — better to
  fail loud at boot than silently mis-route writes.
- **Login CLI fallback**: `lcloud-login` is now real (Telethon phone+code+2FA)
  but kept as the fallback per amendment A2. Wrong-account branch archives
  session files as `data/session.rejected.<unix_ts>.session*` (mode 600) and
  exits with rc=3, matching the same policy the web flow will use in P2.
- **Tests**: `test_crypto.py` covers keypair perms / idempotency / sign /
  verify / payload layout / sha256 streaming. `test_db.py` covers migration
  applies, FTS trigger indexes inserts, idempotent re-run, owner ensure
  end-to-end. `test_health.py` now isolates the lifespan against a tmp
  data dir via env vars + `get_settings.cache_clear()` — patching the
  module attribute fails because callers do `from lcloud.config import
  get_settings` and hold a live reference; clearing the lru_cache on the
  original function works.

## P2 — userbot commands + magic-link auth

Done as **web-based Telegram login** per amendment A2 (saved-messages
magic-link from §8 dropped).

- **Token-bucket rate limiter** (`lcloud/utils/rate_limit.py`): in-process,
  thread-safe, per-key buckets that refill linearly. The /auth/telegram/*
  endpoints share a single 5-token / 5-min bucket per remote IP.
- **JWT cookie** (`lcloud/auth/{jwt_utils,cookies,deps}.py`): HS256, 32-byte
  secret in `data/keys/jwt.secret` (mode 600), generated alongside the
  Ed25519 keypair on first boot. Cookie `lc_session` is HttpOnly +
  SameSite=Strict; `Secure` flag is env-driven (`LC_COOKIE_SECURE`,
  default false because dev runs on plain http://127.0.0.1; flip to true
  once HTTPS is fronted). Payload includes `auth_epoch` (per-owner
  monotonic from `auth_state`); `require_admin` dependency rejects
  `epoch_mismatch` so a future `/revoke` can invalidate live cookies by
  bumping the row.
- **Stale stubs purged**: `types-pyjwt==1.7.1` (PyJWT v1 stubs) was active
  alongside PyJWT 2.10.0, making mypy think `jwt.encode()` returned bytes.
  PyJWT 2.x ships its own `py.typed`, so the third-party stubs are now
  removed from `pyproject.toml` dev deps.
- **UserbotManager** (`lcloud/userbot/client.py`): single-instance Telethon
  client wrapped in a state machine
  `NO_SESSION ↔ CODE_SENT ↔ PWD_NEEDED → AUTHORIZED`. All public coroutines
  serialized by `asyncio.Lock`. Degraded-mode start: if `TG_API_ID/HASH`
  are unset, `start()` no-ops and `is_started` stays false (so `/auth/state`
  reports `userbot_started=false` and operators know setup is incomplete
  rather than seeing a confusing connect error at boot).
- **Wrong-account branch** (`_reject_session`): on `me.id != LC_ADMIN_TG_ID`,
  `client.log_out()` runs first (invalidates the auth_key at Telegram),
  then session files are renamed to `data/session.rejected.<ts>.session*`
  (mode 600), and `data/session.rejected.<ts>.json` is written with
  `{got_user_id, expected_user_id, ts, note}` so the operator can audit
  what happened. A fresh empty client takes the place of the rejected one
  so the next attempt starts from `NO_SESSION`.
- **Order of log_out vs archive**: log_out FIRST, then archive. Reasoning:
  log_out clears the *local* SQLite session db — the archived `.session`
  file ends up empty/cleared. The forensic value is in the JSON sidecar
  (got/expected user_id + ts), not the DB content. The archived `.session`
  file persists as a timestamp marker. This makes the archive harmless
  even if it's later read by an attacker (auth_key is already revoked at
  Telegram).
- **Auth router** (`lcloud/api/auth.py`): GET `/auth/state`, POST
  `/auth/telegram/{start,code,password,cancel}`, POST `/auth/logout`.
  All TG-flow endpoints go through `_enforce_rate(request)`. Pydantic body
  validation runs BEFORE rate limiting (FastAPI default), so a malformed
  body returns 422 without consuming a token — that's the right behavior
  for form errors.
- **Admin owner discovery**: in V1 there is exactly one admin owner
  (single-admin invariant). `_resolve_admin_owner_id()` does
  `SELECT * FROM owners WHERE role='admin' LIMIT 1` rather than mapping
  TG user_id → owner_id. The two are linked implicitly via
  `LC_ADMIN_TG_ID`. Multi-admin in the future would need an explicit
  `tg_user_id` column on `owners`; not adding it speculatively.
- **Test coverage**: `test_jwt.py` (perms, idempotency, roundtrip, expiry,
  wrong-secret), `test_rate_limit.py` (capacity, refill, independent keys),
  `test_userbot.py` (state machine + 2FA + wrong-account archive +
  degraded mode, all with a hand-rolled `FakeTelegramClient` so no real
  Telethon network calls), `test_auth_api.py` (full HTTP flow incl.
  cookie issuance + 403 + 429 + cancel). Total 44 tests, all green.

## P3 — clouds (create / list / scan markers)

- **Marker** (`lcloud/userbot/marker.py`): format `LCLOUD1:<pub>:<sig>` per
  goal.md §6, with both halves base64-urlsafe **without padding** (chosen
  for compactness + URL-safety; saves 4 chars vs standard b64 — total
  marker length 138 chars, comfortably under Telegram's 255-char `about`
  limit). Signed payload is `str(chat_id).encode("ascii")` (the marked /
  bot-API-style id, e.g. `-1001234567890`).
- **Cloud creation** (`lcloud/userbot/clouds.py`): single helper
  `create_cloud_chat(client, *, name, signing_key)` runs Telethon's
  `CreateChannelRequest(megagroup=True)` then `EditChatAboutRequest` to
  stamp the marker. Channel id is normalized via `telethon.utils.get_peer_id`
  so the same `chat_id` stored in DB matches what `iter_dialogs()` returns
  later. Returns `(marked_id, marker, channel_entity)` so the channel
  object can be threaded into follow-ups without an extra `get_entity`.
- **Disconnect** = clear `chat.about` + drop DB row. Per goal.md §9 the
  Telegram chat is intentionally NOT deleted (the user may still want
  history). Best-effort on marker clear: if Telegram refuses (lost admin
  rights, etc.), we still drop the row so the UI stops showing the cloud.
- **Dialog scan** (`lcloud/userbot/scan.py`): walk `iter_dialogs()`,
  filter to `Channel + megagroup=True`, fetch `GetFullChannelRequest` to
  read `about`, parse + verify marker against the admin pubkey, upsert
  into `clouds`. Idempotent — running twice updates `name`/`about` but
  doesn't duplicate. Markers with a foreign pubkey or invalid signature
  are logged + skipped, not stored.
- **Scan timing**: fire-and-forget via `asyncio.create_task` (helper
  `schedule_scan`). Triggered (a) at lifespan boot if the userbot is
  already admin-authorized, and (b) after a successful web-login flow
  (`_issue_session_cookie`). The login response does NOT block on the
  scan — could be 30+ seconds for accounts with many dialogs.
- **Auth gating**: `/clouds *` go through `Depends(require_admin)` →
  reads `lc_session` cookie, validates JWT + epoch, returns `owner_id`.
  `POST /clouds` and `DELETE /clouds/{id}` additionally require
  `manager.is_started` AND `is_admin_authorized()` so we don't try to
  create chats with a half-dead Telethon (returns 503 / 409).
- **DELETE 204 quirk**: FastAPI auto-derives `response_model` from the
  return annotation; `-> None` collides with `status_code=204` (which
  forbids a body). Fixed by returning `Response(status_code=204)` and
  setting `response_class=Response` on the route.
- **`UserbotManager.client` accessor**: was `_client` private; exposed as
  a typed property that raises `UserbotNotConfiguredError` if start()
  wasn't called. The clouds router uses this instead of poking internals.
- **`is_admin_authorized()` method**: convenience predicate, used in
  three places (lifespan post-start, login-flow post-cookie, and the
  clouds endpoints' readiness check).
- **Tests**: `test_marker.py` (roundtrip + tampering + garbage rejection
  + 255-char limit), `test_scan.py` (valid marker / wrong pubkey / idempotent
  re-scan, all against a hand-rolled async-iterator stub), `test_clouds_api.py`
  (401 unauthenticated, list/create/delete authenticated, name validation,
  409 when userbot deauths mid-flight). Total now: 59 tests, all green.

## P4 — worker pool + rate limiter

- **WorkerPool** (`lcloud/workers/pool.py`): semaphore-backed concurrency
  cap. Considered the canonical "queue + N background tasks" pattern but
  rejected it: HTTP handlers want the awaited result of a single
  coroutine, not a futures-based "submit and forget", and a Semaphore
  expresses the same constraint with ~30 lines vs ~150. The coroutine
  runs in the caller's task, so cancellation propagates naturally and
  there's no shutdown-drain bookkeeping. Tracks `active` + `high_water`
  for cheap metrics.
- **AsyncTokenBucket** (`lcloud/workers/rate_limiter.py`): classic
  token bucket, but `acquire()` awaits when empty instead of returning
  False. Lock is held only for arithmetic / consumption; the wait
  itself happens outside the lock so concurrent acquirers are properly
  ordered through the asyncio scheduler rather than starving each other.
  Configurable via `LC_MTPROTO_RATE_PER_SEC` (default 20/s) and
  `LC_MTPROTO_BURST` (default 20).
- **FloodWait policy** (`call_with_floodwait_retry`): on a Telethon
  `FloodWaitError`, sleep `seconds + 1` (small jitter) and retry up to
  `max_retries=3`. If Telegram requests a wait longer than
  `LC_MTPROTO_MAX_FLOODWAIT_SEC` (default 300s), surface the error
  instead of blocking the request — caller decides when to retry.
- **Convenience wrapper** `mtproto_call(client, request)`: runs a TL
  request under the global limiter + retry policy. Single source of
  rate-limit policy; existing call sites (`create_cloud_chat`,
  `clear_cloud_marker`, `scan_dialogs_for_clouds`'s
  `GetFullChannelRequest`) now go through it. `iter_dialogs()` is left
  bare — it's a server-streaming generator and Telethon already does
  some FloodWait handling internally for it.
- **Lifespan integration**: `init_worker_pool` + `init_mtproto_limiter`
  added to lifespan after DB bootstrap, before Telethon start. Both
  exposed as singletons (`get_worker_pool`, `get_mtproto_limiter`)
  with `reset_*` helpers for tests. Disposed via `reset_*` on shutdown.
- **Tests**: `test_worker_pool.py` (cap, propagation, recovery,
  singleton lifecycle) + `test_rate_limiter_async.py` (burst,
  refill, fairness, FloodWait retry / exhaust / above-cap-surfacing,
  singleton lifecycle). Total 73 tests, all green.
- **Backward compat**: existing `lcloud/utils/rate_limit.py` (the simple
  per-IP login limiter) is unchanged. The two limiters are deliberately
  separate: one gates inbound HTTP login attempts (synchronous, fail-fast
  with 429), the other gates outbound MTProto calls (async, awaits with
  retry). Mixing them would muddy semantics.

## P5 — file upload / download via web

- **LC1 caption** (`lcloud/userbot/lc1.py`): `LC1:{"o":...,"s":...,"h":...,"t":...}`
  with all binary fields base64-urlsafe-no-padding (consistent with the
  chat marker). Max-size payload (32B sha + 64B sig + 32B pub + max ts) =
  213 chars, well under Telegram's 1024-char caption limit. `parse_lc1_caption`
  is whitespace-tolerant (Telegram clients sometimes append newlines).
- **Upload module** (`lcloud/userbot/files.py`): the trick is that the
  signature payload includes `message_id`, but Telegram only assigns one
  AFTER the message lands. So flow is: `send_file` (no caption) →
  receive `message_id` → compute signature with the now-known id →
  `edit_message` with the LC1 caption. Also: `force_document=True`
  always — V1 is a file manager, not an image gallery; we never want
  Telegram's lossy photo compression.
- **High-level Telethon vs `mtproto_call`**: kept `send_file`,
  `iter_download`, `get_messages`, `delete_messages`, `edit_message`,
  `get_entity` un-wrapped — those are higher-level helpers that do their
  own chunking + per-chunk retry + entity caching internally. The token
  bucket gates only our raw TL requests (`CreateChannelRequest`,
  `EditChatAboutRequest`, `GetFullChannelRequest`). Wrapping high-level
  helpers would either be a no-op (Telethon already retries) or a foot-gun
  (token bucket + Telethon retry would compound waits).
- **Streaming sha256** (`_stream_to_temp_async`): walks the multipart
  body in 1 MiB chunks, hashing + writing to `data/tmp/upload-<uuid>.bin`,
  aborting at `LC_MAX_FILE_BYTES` (default 1 GiB) with HTTP 413.
  Sync write inside async function — per-chunk blocking is ~10 ms which
  is acceptable for V1's traffic profile (single user). Each chunk's
  `await upload.read(...)` yields the loop. Future polish (P9) could
  push write+hash to `asyncio.to_thread` or use `aiofiles`.
- **Worker pool integration**: `pool.submit(upload_file_to_cloud(...))`
  caps concurrent uploads at `LC_MAX_WORKERS` (10). The web request
  awaits the future. Multiple parallel uploads beyond 10 queue at the
  semaphore.
- **Routers**: split into two — `clouds_files_router` mounts under
  `/clouds/{cloud_id}/files` (list / upload — context is the cloud) and
  `files_router` under `/files/{file_id}` (download / delete — context
  is the file id). Same domain, different URL hierarchies, easier to
  reason about than a single mega-router.
- **Download streaming**: FastAPI `StreamingResponse` over an async
  generator that wraps `client.iter_download(message)`. `Content-Disposition`
  uses both ASCII fallback and RFC 5987 `filename*=UTF-8''<percent-enc>`
  so non-ASCII names survive every browser. `Content-Length` is set from
  the stored `size_bytes` so download progress shows correctly.
- **Delete is soft-on-DB / hard-on-Telegram**: `DELETE /files/{id}`
  calls `delete_messages` in TG (best-effort — if we lost admin or the
  message vanished, log + continue) then sets `files.deleted_at = now()`
  in DB. Hard delete in TG is intentional (don't hoard); soft delete in
  DB preserves audit (sha256, signature, message_id) for forensics.
- **Tests**: `test_lc1.py` (caption build/parse, length sanity),
  `test_files_userbot.py` (upload signature verifies against canonical
  payload — full sign-then-verify roundtrip via the FakeTGClient,
  download yields chunks, missing-message → FileNotFoundError, delete
  forwards to TG), `test_files_api.py` (list empty, upload+list+delete
  flow with TG mocked, oversize → 413, unknown cloud → 404, download
  streams correct bytes + headers, all 401-gated when unauthenticated).
  Total: 88 tests, all green.

## P6 — direct TG upload handler

- **Single NewMessage handler** (`lcloud/userbot/handlers.py`) registered
  via `add_event_handler` (not the `@client.on(...)` decorator — the
  decorator confuses mypy strict because Telethon's stub doesn't propagate
  the inner function's type, and `add_event_handler` is the equivalent
  imperative form). Filter is `incoming=True, outgoing=True` because the
  userbot account IS the admin's account: dropping a file from the admin's
  mobile TG client lands as outgoing on the userbot's side.
- **Filter chain inside handler**: chat is a tracked cloud (DB lookup) →
  caption doesn't already start with `LC1:` → media is a Document (photos
  silently skipped — Telegram compresses photos and the cloud contract
  is exact-bytes preservation). Each early exit returns a short status
  string for tests / observability, never throws.
- **Size cap before download**: `Document.size` is supplied by Telegram in
  the message metadata; we reject oversize files BEFORE downloading them
  (delete the message + Saved-Messages alert). Saves bandwidth on the
  pathological case.
- **Hash without persisting**: `iter_download` chunks pass through sha256
  without ever hitting our disk. The Telegram message itself is the
  canonical store, so we don't need a local copy. RAM stays flat at
  the chunk size (~512 KiB).
- **Size-mismatch defense**: if Telethon's reported `Document.size` and
  the bytes actually streamed disagree, refuse to sign. Otherwise we
  could end up with a valid signature attesting to wrong content.
- **Caption-edit failure → don't persist**: if `edit_message` fails after
  hashing, we don't write the DB row. Result: we still re-process this
  message on the next handler trigger / scan (caption stays empty so the
  LC1-skip filter doesn't trip). Better than a row pointing at a message
  with no marker.
- **Saved-Messages alert format**: human-readable single line, no JSON
  payload. The admin reads these in TG, not via API.
- **Hooked into lifespan**: `_post_login_scan_if_authorized` (renamed
  internally — still kept the function name for back-compat with
  `lcloud/api/auth.py`) now (a) attaches the handler and (b) schedules
  the dialog scan. So both fire after either lifespan boot
  (already-authorized) or fresh web-login (cookie-issued).
- **Tests**: `test_ingest_handler.py` covers happy path (sig verifies
  end-to-end against canonical payload), LC1-skip, non-cloud-skip,
  oversize → delete + alert, non-document skip, size-mismatch refusal,
  caption-edit failure → no row.

## P7 — tags

- **CRUD on `/tags`** (`lcloud/api/tags.py`): GET (list), POST (create
  with unique-name-per-owner, 409 on duplicate), PATCH (partial update,
  conflict-detected on rename), DELETE (cascade drops `file_tags` rows
  via `ON DELETE CASCADE` — already in the schema from P1).
- **Per-file assignment via `/files/{id}/tags`**: PUT replaces the entire
  tag set (deterministic — diff vs current, INSERT additions and DELETE
  removals); GET returns the file's tags. Unknown tag ids in the PUT
  body return a 404 listing the missing ids — easier to debug than a
  generic 422.
- **Owner scoping**: every query is gated on `Tag.owner_id == owner_id` /
  `File.owner_id == owner_id`. The auth dep returns the owner id directly
  so this is one extra `WHERE` clause per query.
- **Color validation**: lightweight regex (`#hex` 3-8 chars OR a CSS
  named-colour token). Strict enough to reject obvious garbage, lax
  enough to accept anything the UI might pass through.

## P8 — search

- **Single endpoint `/search`** (`lcloud/api/search.py`): combines name
  match (FTS5) + cloud filter + tag intersection + pagination. Composing
  these in one query rather than three round-trips keeps the index hits
  cheap.
- **FTS5 input sanitisation** (`_fts_quote`): each search term is wrapped
  in `"..."` and suffixed with `*` for prefix match, then joined by
  `AND`. Quotes inside the input are stripped first to avoid breaking
  FTS5 syntax. Matches the iOS-Files-style "type a substring, see hits"
  expectation in goal.md §11.
- **Tag intersection** is `HAVING COUNT(DISTINCT file_tags.tag_id) = N`
  rather than a chained JOIN — the latter blows up on N tags. The HAVING
  approach handles 1..N tags uniformly, with a single grouped query.
- **Soft-deleted files excluded**: every `/search` result, like every
  `/clouds/{id}/files` result, filters `deleted_at IS NULL`.
- **Tests** (`tests/test_tags_search_api.py`): 11 cases covering tags
  CRUD lifecycle, color validation (422 on garbage), per-file assignment
  + replace-with-empty, unknown-tag/file 404s, cascade-on-tag-delete,
  FTS prefix match, single-tag filter, multi-tag intersection, combined
  q+tags filter, 401 on unauthenticated. Total now: 106 tests, all green.

## P9 — UI polish

- **Stack**: React 18 + Vite 5 + TS strict + Tailwind 3 + TanStack Query
  + lucide-react. Pinned versions in `web/package.json`. Skipped
  shadcn/ui — wrote three minimal Tailwind-styled primitives
  (`Button`/`TextField`/`Modal`) instead. The shadcn workflow generates
  ~30 component files for what we need 3 of, and pulling in Radix would
  add ~200 KB of JS for primitives we don't use.
- **Vite ↔ FastAPI**: in dev, Vite runs on `127.0.0.1:8788` and proxies
  `/auth`, `/clouds`, `/files`, `/tags`, `/search`, `/health` to
  `127.0.0.1:8787`. In prod, FastAPI mounts `web/dist/assets/*` as
  static files and serves `index.html` at `/` plus a SPA-fallback at
  `/{full_path:path}` so client-side routing survives hard-refresh.
  The fallback skips known API prefixes so a typo in an API call
  returns 404 rather than the SPA HTML (which would be a confusing
  silent failure for fetch callers).
- **Auth gate**: `useAuth` polls `/auth/state` every 4 s while
  unauthenticated; flips to "no refetch" once authorized. The whole UI
  shell is conditional on that single boolean — keeps the login
  screen → app transition trivial. After successful login, React Query
  refetches and the app pivots to `<Sidebar/> + <FilesPanel/>`.
- **Login flow as a state machine inside one component** (`LoginScreen.tsx`):
  `phone → code → password`, with `Cancel` resetting back to `phone` and
  hitting `/auth/telegram/cancel`. Errors from the API are rendered as
  red alerts under the form. The whole component is ~140 lines because
  there's no router — just `useState<Stage>`.
- **API client** (`web/src/api/client.ts`): typed wrapper around
  `fetch` with `credentials: "same-origin"` (relies on the `lc_session`
  HttpOnly cookie). One special case — `files.upload` uses `XMLHttpRequest`
  rather than `fetch` because `fetch` has no upload-progress event in
  browsers; XHR fires `upload.onprogress`. Without that, multi-GB
  uploads through the UI would look frozen.
- **Drag-n-drop + `<input type=file multiple>`**: same code path
  (`handleFiles(FileList | File[])`), so dragging onto the panel and
  clicking "Upload" share progress tracking + error display + cache
  invalidation. Progress entries live in component state, surviving
  the upload then fading on completion (or staying with a red
  `❌ <reason>` line on error).
- **Search vs list**: when `cloud_id` is null, or the user typed a
  query, or any tag filter is active → `/search` (FTS5 + tag intersection)
  otherwise `/clouds/{id}/files` (cheap, indexed). Picked at query-key
  level so React Query caches each variant independently.
- **Tags**: `TagFilterBar` doubles as the create-tag entry point
  (top-right `+ новый тег` link → modal with palette of 12 colours,
  CSS-style `<input type=color>` for background, 20-icon picker over
  lucide-react names + live preview chip). Per-file tag edit is a
  flat list of toggle-chips inside each file card — click to add/
  remove, immediate PUT under the hood.

- **systemd unit** (`lcloud.service`): `Type=simple`, single
  `python -m uvicorn lcloud.main:app` invocation,
  `EnvironmentFile=/root/LCloud/.env` so the same `.env` the dev flow
  uses drives prod. `--no-access-log` because uvicorn's default log

## Production deployment notes (post-V1)

### Reverse-proxy on shared host (luxiva + LCloud)

The host already runs the `luxiva` shop stack — `shop-nginx` container
owns 80/443. We piggyback on it instead of installing Caddy/another
nginx:

- `LC_HOST=0.0.0.0` so the FastAPI listener accepts connections from
  the docker-bridge gateway (172.18.0.1). UFW limits inbound on 8787
  to `172.18.0.0/16` (the `luxiva_default` bridge), so the service is
  not exposed publicly on 8787 — only via shop-nginx → 443.
- `proxy_headers=True, forwarded_allow_ips="127.0.0.1,172.18.0.0/16"`
  on uvicorn so per-IP rate limiting in `/auth/*` sees the real client
  IP rather than nginx's bridge address.
- `LC_COOKIE_SECURE=true` in `.env`. Tests inject
  `LC_COOKIE_SECURE=false` via fixtures because `httpx`'s cookie jar
  refuses Secure cookies over plain `http://testserver`.

### nginx HTTPS server block (added by `scripts/setup-tls.sh`)

The TLS server block carries the file-transfer tuning that goal.md §2
demands:

- `client_max_body_size 1100m;` (1 GiB hard limit + multipart slack)
- `proxy_request_buffering off; proxy_buffering off;` — streams both
  directions so a 1 GiB upload doesn't first land on nginx's disk
- `proxy_read_timeout 3600s;` etc. — multi-GB transfers are minutes,
  not seconds
- standard HSTS / X-Frame / Referrer-Policy headers, ssl_protocols
  TLSv1.2/1.3 (matching luxiva's vhost so the operator has one TLS
  posture across the host)

### TLS issuance

Let's Encrypt via webroot challenge against `/root/luxiva/nginx/certbot`
(same path the shop already uses, so renewals share infrastructure).
`certbot.timer` is enabled host-wide and runs the renewal twice daily;
the LCloud cert renews alongside the shop cert with no extra cron.

### `scripts/setup-tls.sh` design

Idempotent end-to-end installer:
1. DNS sanity check — refuses to run if the domain doesn't resolve to
   this server (would break HTTP-01 anyway, but explicit error is
   friendlier than certbot's stack trace).
2. `certbot certonly --webroot ...` once, skipped on re-run if cert
   already exists.
3. Append HTTPS server block to the shop-nginx config (separate
   markers `# >>> tg-lcloud TLS` ... `# <<<` so it's removable).
4. `nginx -t` inside the container, then `nginx -s reload`.
5. Restart `lcloud.service` to pick up `LC_COOKIE_SECURE=true` etc.

### `lcloud.service` — switched to the `lcloud` console-script

The initial unit ran `python -m uvicorn` directly with `--host 127.0.0.1`
hardcoded, which (a) didn't read `LC_HOST` from settings and
(b) needed `EnvironmentFile=` for `.env`, but systemd's parser doesn't
allow inline `# comment` after values (which pydantic-settings does
allow). Both problems disappear by calling `/root/LCloud/.venv/bin/lcloud`
— the entrypoint reads the dotenv via pydantic and binds based on
settings.

## Production amendment: full process isolation (post-deploy)

The user wanted LCloud entirely independent of the existing
`shop-nginx` container — restarting one must never touch the other. We
moved away from the host-based virtual-hosting on shop-nginx:443:

- **Dedicated Caddy** on `127.0.0.1:8443` (apt package, systemd unit
  with a `User=root` override so it can read `/etc/letsencrypt/live/...`
  without ACL gymnastics). Caddyfile uses `auto_https off` + manual
  `tls /path/to/cert /path/to/key`, so Caddy doesn't try to obtain
  its own cert (would clash with shop-nginx on :80).
- **All tg-lcloud blocks removed from shop-nginx**. shop-nginx is now
  the luxiva-only proxy on 80/443; Caddy is the LCloud-only proxy on
  8443. Neither config knows about the other.
- **LCloud back on `127.0.0.1:8787`** (private). The previous
  `LC_HOST=0.0.0.0` + ufw-allow-from-bridge workaround is gone; Caddy
  reaches uvicorn on plain loopback.
- **`LC_PUBLIC_BASE_URL=https://tg-lcloud.duckdns.org:8443`** so any
  server-side URL generation includes the port.
- **Cert renewal hook**:
  `/etc/letsencrypt/renewal-hooks/deploy/lcloud-caddy-reload.sh`
  reloads Caddy whenever certbot renews `tg-lcloud.duckdns.org`. The
  shop's own renewal hook (for shop-nginx) is untouched.
- **Caddy upload tuning** matches the goal.md §2 limit: `request_body
  { max_size 1100MB }`, global `read_body 1h`, `flush_interval -1` on
  the reverse_proxy so 1 GiB streams in/out without buffering.
- **Trade-off accepted**: the URL now carries `:8443`. Visiting plain
  `https://tg-lcloud.duckdns.org/` will land on shop-nginx's default
  vhost (which doesn't match anymore), so the user must use the
  port-explicit URL. This is the cost the user explicitly chose for
  full isolation.
